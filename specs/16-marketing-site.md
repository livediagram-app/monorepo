# 16, Marketing site

The marketing app (`apps/marketing`) is the public landing site served at `/` (everything the router doesn't hand to the live editor's routes, `/api/*`, or `/telemetry`, see [08-router-app](08-router-app.md)). Static Next.js export, no SSR. It is the top of the acquisition funnel: its only job is to explain what livediagram is and send visitors to `/new`.

## Golden rule: claims map to shipped features

**Every feature claim on the marketing site must correspond to something a visitor can actually do today.** No aspirational copy, no roadmap dressed as present tense.

- If a feature is not built (teams/permissions, true concurrent/CRDT editing, see [02-prototype-scope](02-prototype-scope.md)), it does **not** appear on the landing page, even softened.
- When the editor gains or loses a feature, update the landing page in the same change. Drift between the page and the product is a bug.
- Counts (templates, themes, shapes) must match the source of truth ([09-canvas-and-command-palette](09-canvas-and-command-palette.md), `apps/live/lib/templates.ts`, `apps/live/lib/themes.ts`). As of this spec: **30 templates** (10 default + 20 extra, grouped into Mind maps / Flowcharts / Hierarchies / Agile / Project Management / Strategy / Design / Technical categories in the picker, including three mind-map styles (radial / tree / bubble), four flowchart styles (basic / swimlane / decision tree / approval workflow) plus a data-flow diagram, Mobile / Laptop / Slide-deck wireframes that pair with the device-frame shapes, a Flywheel momentum loop, a Logo design starter, a Gantt project-planning chart, a Live card greeting-card lockup, and the technical System architecture / Database schema / Sequence diagram plus an Impact-vs-Effort Prioritization matrix), **26 themes** (12 default + 14 extra, the extras including a Dark category and five multi-colour "rainbow" themes — see [spec/29](29-multicolour-themes.md)). Both counts are pinned by `apps/live/lib/templates.test.ts` + `themes.test.ts` so spec drift surfaces as a test failure.
- Be honest about trade-offs where it builds trust: realtime is **last-write-wins**, not CRDT, say so rather than implying conflict-free merging.

## Structure

Single landing page (`app/page.tsx`) with a sticky header, hero, themed feature sections, a closing CTA, and a footer. Feature sections group related capabilities (rather than two large catch-all grids) and alternate the `tinted` background for rhythm.

0. **Header** (`components/Header.tsx`), sticky bar with the `Brand`, the **social share row** (see below), a quiet **Explorer** utility link → `/explorer` (ghost-styled a notch below Share, folder icon + a label that hides below `sm:`; opens the editor's full library, which is open to guests), and the primary CTA → `/new` ("Start drawing"). Order left→right: Share, Explorer, Start drawing.
1. **Hero** (`components/Hero.tsx`), headline ("A picture tells a thousand words, tell your story"), conversion-focused subhead, a single primary CTA → `/new` ("Start drawing") with a no-sign-up note, and the interactive `HeroIllustration`. The illustration is a **three-window sliding stage**: three editor windows ride a track (`.hero-track`), each `card%` of the stage with a gutter (≈68% on `sm+`, ≈88% on mobile so the windows stay legible). The windows are (1) a **flowchart** — shared, a teammate **cursor**, and the **theme-recolour beat** (it is the **only** window that _animates_ a recolour, blue→green, with its **canvas tinting in step** via `.hero-theme-canvas`); (2) a **mind map** — shared, on a **violet** theme, with a **laser pointer** that rings the top-left node then moves to the bottom-right node and rings it (`hero-laser-*`); (3) a **release timeline** — **private** (amber Private badge, just the owner avatar, no collaborators or pointer), on an **amber** theme. Each window sits on its own themed canvas tint (matching its shapes); only the flowchart's tint animates. All share the editor chrome (header with the `Brand` mark — icon + `livediagram` wordmark, no brackets — the Shared/Private badge, presence; dot-grid canvas + palette; bottom tab bar whose colour-coded tabs are **relevant to that window's diagram**, plus the active-tab ellipsis and the toolbelt the page advertises: search, shortcuts, settings, dark mode — the toolbelt hides on mobile, where it would clash with the tabs in the narrower windows) and the same pure-CSS `hero-*` build keyframes. **Only the centred window animates**; its build replays each time it reaches centre (the diagram group is keyed on `playing`), while peeking windows render settled (`.hero-static`), blurred + faded, with the stage edges masked so they fade out rather than hard-clip. The stage **auto-advances every 16s** and **centres a window when clicked** (React state drives `translateX`; the timer resets on interaction). With JS off it renders the first window centred; reduced-motion settles every build, holds the canvas tint, and hides the laser.
2. **Simple by design** (`#why`): the positioning lead-in. Tagline "Simple by design, powerfully deep" plus the easy-yet-powerful duality (start in one click, looks simple/runs deep, multiplayer with no setup, works on any device) and the easiest-start features: templates, themes, and keyboard shortcuts (the catalogue from the editor's shortcuts dialog + its per-device toggle; the copy now also covers the per-element add keys, the tool-switch keys, and the Cmd-hold palette key badges, not just undo/redo/delete/pan). The duality cards use light icon badges; the templates / themes / shortcuts cards keep their canvas-style mocks. (Device frames + the wireframe templates still ship; they're shown through the templates and the hero illustration rather than a dedicated card.)
3. **Collaboration** (`#collaboration`, tinted): the shared-canvas story. Editor/view-only links, live presence, selection glow, realtime edits, comments, the laser pointer, and stop-sharing/revoke.
4. **Use-case carousel** (`components/UseCaseCarousel.tsx`): a deliberately different layout from the `FeatureGrid` sections — a dark band with one large featured use case, two smaller cards beside it, and prev/next arrows + dot indicators to rotate through the rest; clicking a side card promotes it to featured. Sits high up (right after Collaboration) as a "what could you build" hook before the deeper feature sections. Illustrative only (the cards don't link anywhere — the page CTAs handle conversion), and every use case maps to a shipped template / shape capability so the golden rule holds. It's one of the page's `'use client'` boundaries (see Social sharing).
5. **Refine** (`#refine`): working the canvas: multi-select, group, lock, format painter, folders, the light/dark UI toggle (per-device; see [07-live-app](07-live-app.md)), and the minimal panel layout (opt-in preference that swaps the floating side panels for the mobile-style compact dock + pop-out panels; always on for mobile, per-device; see [20-user-preferences](20-user-preferences.md)). (The earlier `#features` "Draw" section that itemised shapes + arrows was retired; its shapes/arrows coverage now lives in `#versatile` as dedicated shape, arrow, and freehand cards rather than a standalone section.)
6. **Tabs** (`#tabs`, tinted): unlimited tabs per diagram, link elements across tabs, copy a tab into another diagram (a one-time copy, not a live sync), lock a tab, reorder and theme-colour-code, and group tabs into named, collapsible folders along the tab bar for large diagrams (see [30-tab-folders](30-tab-folders.md)). Tabs have no manual colour or order field: order is array position (drag), colour is derived from the tab's theme accent.
7. **Diagrams you can rely on** (`#reliability`): autosave, undo/redo, activity log + revert, durable save (survives a refresh), account sync, and global search (jump to any diagram / folder / tab / element by name; see `apps/live/components/SearchPanel.tsx`). Note: undo/redo is capped at `HISTORY_LIMIT = 3` steps, so the copy says "recent edit", not unlimited.

7a. **Privacy callout** (`<PrivacySection />`, between `#reliability` and `#versatile`): a free-standing band emphasising the "no third-party tracking pixels" promise as a confidence beat before the breadth-of-features section that follows. Mounted directly from `page.tsx`; its own component rather than a `Section`, so it carries its own background and layout. The MIT / no-servers / no-tracking foundations cards in section 9 below repeat the message in card form.

8. **As versatile as your ideas** (`#versatile`, tinted): the breadth of one canvas — the optional **AI assistant** (lead card: Build / Clean / Ask / Review modes over the selection or active tab; opt-in and hidden unless the api worker has `OPENAI_API_KEY`, so self-hosters bring their own key; see [25-ai-assistance](25-ai-assistance.md)), images (drag / drop / paste a PNG / JPEG / WebP / GIF, reuse from a per-owner gallery; see [19-images](19-images.md)), **shape rotation** (rotate handle, 15° snapping, Shift for free, rotation-aware arrow anchoring; see [09](09-canvas-and-command-palette.md)), the shape library (ten core shapes + browser / monitor / laptop / phone / tablet device frames, click-to-drop or drag-to-size with neighbour snapping, see [09](09-canvas-and-command-palette.md) Draw-to-size), arrows (straight / curved / angled with draggable curve + elbow handles, thickness, arrowhead size, labels, pinned ends), freehand (the Pencil tool, F key, with the opt-in shape-recognition mode that snaps a rough rectangle / circle / diamond / line into a clean primitive; see [09](09-canvas-and-command-palette.md) Pencil), per-element notes, border styling (strength + solid / dashed / dotted + corner radius), and the canvas backdrop (a dozen background patterns: grid, lines, crosshatch, waves, stars, … or blank). This is where the retired "Draw" section's shapes/arrows coverage now lives, expanded with the arrow-handle and freehand work.
9. **Open and honest** (`#foundations`): MIT license, self-host, no third-party tracking pixels (the only usage data we collect is anonymous, first-party product events; the public `/telemetry` dashboard shows exactly what we measure, see [22-telemetry](22-telemetry.md)). See also [03-open-source-and-business-model](03-open-source-and-business-model.md).
10. **Closing CTA** (`#get-started`), "no sign-up wall" message + `/new`.

The primary CTA reads **"Start drawing"** everywhere (header, hero, closing CTA), keep it consistent.

Reusable building blocks: `Section` + `FeatureGrid` (`components/Section.tsx`), `Header`, `Footer`, `Brand` (from `@livediagram/ui`). Add new feature cards by editing the `items` arrays in `page.tsx`, do not fork the grid.

## Social sharing

Word of mouth is part of the acquisition funnel, so the page lets a happy visitor pass it on in one click. `components/ShareButtons.tsx` renders a **"Share" button** (share-nodes icon) in the header, just left of the "Start drawing" CTA. Clicking it opens a small **popover** listing the share targets (X, LinkedIn, Facebook, WhatsApp, email) plus a **Copy link** action. The popover closes on outside click or Escape; the button keeps its icon at every width and shows the "Share" label from the `sm` breakpoint up.

- **Share targets are plain anchor links** to each network's public share-intent URL (`twitter.com/intent/tweet`, `linkedin.com/sharing/share-offsite`, `facebook.com/sharer`, `wa.me`, `mailto:`). No SDKs, no tracking pixels (keeps the "no tracking pixels" claim honest under [03](03-open-source-and-business-model.md)). They open in a new tab (`target="_blank"` + `rel="noopener noreferrer"`) and close the popover on click.
- **Copy link** uses the Clipboard API to copy the canonical URL and flips its row to "Link copied" for a beat.
- The shared URL is the canonical production site (`https://livediagram.app`) and the share text mirrors the product's one-line description. Both live as constants in the component, update them together if the positioning copy changes.
- `ShareButtons`, the `UseCaseCarousel` (a stateful featured/side rotation), and the `HeroIllustration` (the click-to-centre three-window stage) are the `'use client'` boundaries on the page; everything else stays a server component for static export.

**Feature-card illustrations.** Every feature card carries a small animated mock of the real editor surface it describes (`components/FeatureArt.tsx`), passed via the card's `art` slot. Like the hero, the motion is **pure CSS**, shared keyframe classes (`fa-*`) live in `app/globals.css`, so it works under static export with no JS and settles to a finished frame under `prefers-reduced-motion`. The mocks must stay faithful to how the feature actually looks (brand-blue rounded shapes, dot-grid canvas, presence avatars, the explorer / share / activity panels); when a feature's UI changes, update its art. This is part of the golden rule above, an illustration is a claim too.

## Tone & brand

- Product name **livediagram** (lowercase). Brand colour sky-blue `#0EA5E9` (`brand-500`), see [01-color-scheme](01-color-scheme.md).
- Fast, clean, structured. Plain and confident; not cutesy, not enterprise-jargon.
- Positioning: multiplayer-from-the-start diagramming for teams who think visually, between casual whiteboards and heavyweight suites (see [00-purpose](00-purpose.md)).

## Auth messaging

The canvas works without signing in, and that stays the headline ("no sign-up wall", [04-auth-and-guest-access](04-auth-and-guest-access.md)). Do not advertise account-only benefits (cross-device sync, teams) until they are shipped and verified end-to-end.

## Not yet present

- No `/pricing` page. The hosted product is free for everyone with no tiers ([03](03-open-source-and-business-model.md)), so there's nothing to price.

## SEO and metadata

The marketing site is the indexable surface for the product, so its `<head>` carries the search and social signals. The editor's routes and the API at `/api/*` are not optimised for organic search (the editor lives behind client-side bootstraps and has nothing useful to index).

- `metadataBase`: `https://livediagram.app`. Lets relative paths in metadata resolve into absolute URLs (the canonical, OG image, sitemap entries).
- `alternates.canonical`: every static page declares its own canonical (`/`, `/faq`, `/terms`, `/privacy`). Stops query-string variants and the `www` mirror from competing with the canonical form.
- `<html lang="en-GB">` on the root layout. Matches the OG locale + JSON-LD `inLanguage` so every locale signal across the page agrees. The earlier broader `lang="en"` worked but reads inconsistent next to OG's specific `en_GB` declaration.
- `openGraph`: title, description, url, siteName, locale (`en_GB`), type (`website` for the landing page, `article` for FAQ/legal/alternatives). Mirrors the page-level title/description so social cards match the actual head. Article-type subpages with a tracked revision date (`/terms` + `/privacy` via `LEGAL_LAST_UPDATED`, `/alternatives` + `/alternatives/<slug>` via `ALTERNATIVES_LAST_UPDATED`) also emit `article:modified_time` via `subpageMetadata({ modifiedTime })`, the OG meta tag Google + social previews read for the freshness signal on article pages. FAQ omits it because the content moves with product releases, not on a tracked schedule. Card image is generated by `apps/marketing/app/opengraph-image.tsx` via `next/og`'s `ImageResponse`; Next.js's file convention pre-renders it to a static 1200x630 PNG at build time and ships it in `out/`, so the asset has no runtime dependency.
- `twitter`: card `summary_large_image`, title, description. `apps/marketing/app/twitter-image.tsx` re-exports the OG handler so the Twitter card and the OG card stay byte-identical. Copy is identical to the OG / page-level title and description.
- `robots`: explicit `{ index: true, follow: true }` on every page. The hosted product wants indexing; ambiguity about defaults is the kind of thing a future migration silently breaks.
- `app/sitemap.ts`: lists the indexable marketing URLs, `/` + `/faq` + `/terms` + `/privacy` + `/alternatives` + one entry per `/alternatives/<slug>` competitor page (mapped from `ALTERNATIVE_SLUGS` so adding a competitor updates the sitemap automatically, see [21-comparison-pages](21-comparison-pages.md)). The two public operational surfaces, `/telemetry` (the anonymous-events dashboard) and `/status` (the service-status page), are also listed: both are public, indexable (each sets `robots` `index: true` and a self-canonical), and reachable from the footer, so sitemapping them lets crawlers discover them directly rather than depending on footer links. They sit at a lower priority (0.4) than the marketing content because they're transparency pages, not primary landing destinations. `lastModified` is per-page truth, not blanket build-time: `/` and `/faq` use `new Date()` because they change with product releases; `/telemetry` and `/status` also use `new Date()` (the dashboard reflects live event data, the status page reflects the latest deployed state, so both are "fresh as of this build"); `/terms` and `/privacy` carry an explicit `LEGAL_LAST_UPDATED` constant matching the `lastUpdated` prop their `<LegalPage>` JSX advertises; `/alternatives` + the per-competitor pages share an `ALTERNATIVES_LAST_UPDATED` constant. Without that split, every build claims the legal disclaimer changed today, which is misleading to crawlers and dilutes the change-frequency signal for pages that actually move. Next.js generates `/sitemap.xml` at the root from this file.
- `app/robots.ts`: allows the marketing routes, points at the sitemap, and `Disallow`s the live-app routes and `/api/`. The router worker stitches all three apps onto the same hostname, so an unconditional allow would let crawlers waste budget on the editor's auth-walled HTML and the API's JSON responses. The live app also declares `noindex, nofollow` at its root layout (see [07-live-app.md](07-live-app.md)) as the in-page complement; together the two signals tell crawlers to skip the live app entirely.
- **`viewport` export** (Next 15 metadata API): the root layout exports a `Viewport` object carrying `themeColor: '#0EA5E9'` (brand-500, see [01-color-scheme](01-color-scheme.md)) and `colorScheme: 'light'`. The first tints mobile browser chrome (Android Chrome's URL bar, iOS PWA status bar) to the brand, the second declares the **marketing site** is light-mode-only (its own pages have no dark theme). This is about marketing's chrome only — the editor ships a light/dark UI toggle, which the page is free to advertise (see [07-live-app](07-live-app.md)). Both are head meta, not page content, so they don't conflict with the "claims map to shipped features" rule.
- **`app/manifest.ts`**: emits `/manifest.webmanifest` at build time with `name`, `short_name`, `start_url: '/'`, `display: 'standalone'`, the brand theme + background colours, and the existing `/icon.svg` as the icon (which `purpose: 'any'` covers since the SVG is monochrome on a single brand stroke). Surfaces the Android Chrome "Add to Home Screen" prompt + the iOS Safari "Add to Home Screen" landing icon, with no behaviour change for normal page loads. Same `export const dynamic = 'force-static'` opt-in as robots.ts / sitemap.ts so it round-trips through `output: 'export'`. Because `livediagram.app` is one origin (the router stitches the apps by path), this is the **origin's** manifest: the editor (live app) references it (and `/apple-icon`) by absolute path from its own layout metadata, so installing the PWA from the editor gets the same icon + name rather than nothing (see [07-live-app](07-live-app.md)).
- **JSON-LD structured data**: the root layout renders a single `<script type="application/ld+json">` carrying a `@graph` of two schema.org types: `WebSite` (so the brand-name search produces a rich result) and `SoftwareApplication` (so the product is categorised correctly in browse / category-style results). Both schemas restate facts already in the page (name, url, description) and the SoftwareApplication offer is `price: "0"` (truthful: the hosted product is free with no paid tier, see [03](03-open-source-and-business-model.md)). No claims that aren't already true on the surface. Each schema also carries an `image` field pointing at the same OG card (`/opengraph-image`) so Google can pair the brand card with a visual, an `inLanguage` field matching the OG locale (`en-GB`), and a `sameAs` array on the SoftwareApplication that links the canonical GitHub repo (already linked from the footer + the FAQ self-host answer) so the brand's open-source presence is connected to the same entity.
- **FAQ JSON-LD**: `/faq` emits an additional `<script type="application/ld+json">` with schema.org `FAQPage` + a `Question` / `Answer` pair per entry. Unlocks Google's "expandable FAQ" rich result for the page. The JSON-LD `acceptedAnswer.text` is the plain-text form of each answer; entries that render JSX (a link to /privacy in the data-storage question, for instance) carry a parallel `aText` field with the same sentence, sans markup. The FAQ component reuses the existing `FAQS` array as the source so the structured data can't drift from the visible content.
- **Breadcrumb JSON-LD**: every subpage (`/faq`, `/terms`, `/privacy`, `/alternatives`, `/alternatives/<slug>`) emits a `BreadcrumbList` script. Two items for two-level pages (`Home → <Page>`), three items for the comparison detail pages (`Home → Alternatives → <competitor>`). Unlocks Google's SERP breadcrumb display so the search result shows `livediagram.app > FAQ` instead of the bare URL. Rendered via `components/BreadcrumbJsonLd.tsx` so the pages share one source of truth. The landing page (`/`) doesn't get a breadcrumb (it IS home, the trail would be a single entry).
- **ItemList JSON-LD**: the `/alternatives` hub page emits an `ItemList` script whose `itemListElement` array lists every `/alternatives/<slug>` URL in display order. This is the schema.org shape Google expects for a curated index of related pages: it tells crawlers the page is a hub (not a piece of editorial content in its own right), surfaces the linked comparisons as a coherent set rather than scattered backlinks, and can enable carousel-style rich results for the hub itself. Built from the same `ALTERNATIVES` array the visible list + sitemap consume, so adding a competitor updates all three together.

When a new public page lands, add it to both `app/sitemap.ts` and its own metadata block (with `alternates.canonical`). Updating one without the other is a regression.

## Content pages (FAQ, legal)

`/faq` (`app/faq`), `/terms` and `/privacy` (`app/terms`, `app/privacy`) are static routes. Terms and Privacy use the shared `components/LegalPage.tsx` shell; FAQ is a question/answer list. All share the `.legal-prose` styling in `globals.css`. The footer's nav links to Compare (`/alternatives`, the comparison-page hub, see [21-comparison-pages](21-comparison-pages.md)) / FAQ / Terms / Privacy / Telemetry (`/telemetry`, the public anonymous-events dashboard, see [22-telemetry](22-telemetry.md)) / Status (`/status`, the operational dashboard) / Contact (a `mailto:`). The bottom copyright row carries the MIT licence note alongside a Manager Toolkit cross-promo and an author link (tommcclean.me); the footer's nav row also carries a GitHub mark icon link at the end (next to Contact) for visitors who want the source code, with the repo additionally linked from FAQ, the status page, and the `sameAs` JSON-LD in `layout.tsx`. These are plain-English first drafts grounded in what the hosted service actually does (Cloudflare for hosting/DB, Clerk for auth, a localStorage guest id, no trackers/telemetry); they have **not** had legal review, and the contact address (`hello@livediagram.app`) and any governing-law / legal-entity details are placeholders to confirm. Keep the privacy page honest against the data the product really touches (it backs the "no tracking pixels" claim).
